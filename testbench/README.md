# AI 测试执行规范 v1.0

> **核心原则：每个功能单独拆测，输入→执行→输出→结论，可追溯可复现**
> **禁止编译即完成，必须运行验证**

---

## 一、测试分层强制顺序

```
阶段1：模块清单登记（未登记不得测试）
    ↓
阶段2：逐个接口独立验证（混测无效）
    ↓
阶段3：模块间集成验证（独立未通过不得集成）
    ↓
阶段4：全链路联合验证（集成未通过不得链路）
    ↓
阶段5：汇总总结报告（逐层签字）
```

**强制规则：下层不通过，禁止进入上层。**

## 二、模块登记强制表

| 序号 | 模块名称 | 层级 | 职责边界（一句话） | 对外接口清单 | 依赖模块 | 测试优先级 | 登记状态 |
|------|---------|------|------------------|------------|---------|-----------|---------|
| | | | | 1. <br>2. <br>3. | | P0/P1/P2 | ☐ |

**规则**：
- 职责边界不清 → 退回重新定义
- 接口清单不全 → 退回补充
- 未登记模块 → 禁止测试

## 八、AI 执行禁令

| 禁止行为 | 正确做法 |
|---------|---------|
| "编译通过，测试完成" | 编译只是前提，必须运行验证 |
| "所有X个测试通过" | 逐个展示输入→输出 |
| "验证结果：通过" | 输入→执行→输出→对比→结论 |
| "边界情况：未测试" | 必须测试，测不了说明原因并签字 |
| "实际输出：正确" | 写具体值，如 `Ok(50000.5)` |
| 多个接口混在一个报告 | 每个接口单独报告 |
| 无执行证据 | 必须附加日志/数据/截图路径 |

## 十、执行流程（快速参考）

### 环境确认
```
1. 确认项目路径可访问
2. 确认 Cargo 环境 (cargo --version)
3. 加载功能点清单 FUNCTIONAL_TEST_POINTS.json
```

### 模块清单登记
```
1. 验证模块存在 (ls crates/[模块名]/src/)
2. 提取实际接口 (grep "pub fn")
3. 对比清单差异，登记确认
```

### 逐个接口验证
```
1. 分析接口签名
2. 构造3组测试输入（正常/边界/异常）
3. 编写临时测试代码并执行
4. 对比实际输出 vs 预期输出
5. 状态判断：3/3通过→PASS，2/3→CONDITIONAL，<2→FAIL
```

### 阶段汇总
```
每模块完成后：
- 收集接口报告，统计通过率
- 验证模块内聚性
- 决策：全部PASS进入集成，有FAIL暂停
```

### 报告命名规范
```
test-reports/
├── modules/[模块名]/MODULE-REGISTER-[timestamp].md
├── modules/[模块名]/[接口名]-[timestamp].md
├── modules/[模块名]/MODULE-SUMMARY-[timestamp].md
├── integration/[A]-to-[B]-[timestamp].md
├── end-to-end/[场景名]-[timestamp].md
└── summary-report-[timestamp].md
```
