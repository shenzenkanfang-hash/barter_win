# Trading System Business Architecture Overview

交易系统业务全景图 - 业务视角简化版

---

## 核心原则

> **数据驱动业务，事件触发决策，风控把关执行**

---

## 第一层：业务数据流

```
市场数据进入 → 数据标准化 → 指标计算 → 信号生成 → 交易决策 → 风险检查 → 订单执行 → 成交确认
     ↑                                                                                          ↓
     └──────────────────────────────── 状态反馈（持仓/资金变化）────────────────────────────────────┘
```

| 阶段 | 输入 | 输出 | 关键问题 |
|------|------|------|---------|
| 数据采集 | 交易所原始报价 | 标准化K线 | 延迟多少？数据缺失怎么办？ |
| 指标计算 | K线数据 | 技术指标值（趋势/波动/强度） | 计算周期？指标失效怎么办？ |
| 信号生成 | 指标值 | 交易信号（买/卖/持有） | 信号冲突怎么办？频率限制？ |
| 交易决策 | 信号+当前持仓 | 交易指令（开/平/加/减） | 仓位多少？价格多少？ |
| 风险检查 | 交易指令+账户状态 | 通过/拒绝/修改 | 拒绝后怎么办？ |
| 订单执行 | 合规指令 | 成交回报 | 部分成交？超时？失败？ |
| 状态更新 | 成交结果 | 新持仓/新资金/新盈亏 | 数据一致性？ |

---

## 第二层：五层业务架构

交易系统分为五个核心业务层，每层有明确的职责边界：

```
┌─────────────────────────────────────────────────────────────────┐
│                      数据层 (b_data_*)                           │
│  职责：从交易所获取市场数据，提供标准化的K线                      │
│  输入：交易所原始报价                                             │
│  输出：标准K线数据（open/high/low/close/volume/timestamp）       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      引擎层 (f_engine)                           │
│  职责：事件驱动协调，管理整个处理流程的节奏                        │
│  输入：K线数据                                                    │
│  输出：触发各层处理                                                │
│  关键：单品种串行，不同品种独立并行                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      信号层 (c_data_process)                     │
│  职责：计算技术指标，将数据转化为可交易的信号                      │
│  输入：K线数据                                                    │
│  输出：指标值（Z-score、TR比率、RSI、布林带颜色等）               │
│  内部状态：历史指标缓存（计算需要历史数据）                        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      策略层 (d_checktable)                        │
│  职责：根据信号和当前状态，做出交易决策                            │
│  输入：指标信号 + 当前持仓状态                                     │
│  输出：交易指令（开仓/平仓/加仓/减仓/对冲）                       │
│  关键：WAL日志保证崩溃可恢复                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      风控层 (e_risk_monitor)                      │
│  职责：审核交易指令，确保符合账户和规则要求                        │
│  输入：交易指令 + 账户状态                                        │
│  输出：通过/拒绝/修改后的指令                                     │
│  关键：前置检查，不过则不执行                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                      执行层 (b_data_mock / b_data_source)        │
│  职责：将指令发送到交易所，等待成交回报                            │
│  输入：风控通过后的指令                                           │
│  输出：成交结果（成功/部分成交/失败）                             │
│  关键：沙盒与实盘接口一致                                        │
└─────────────────────────────────────────────────────────────────┘
```

### 层间数据传递规则

| 上游层 | 下游层 | 传递的数据 | 约定 |
|--------|--------|-----------|------|
| 数据层 | 引擎层 | K线数据 | 格式固定，时间连续 |
| 引擎层 | 信号层 | 触发信号 | 每K线触发一次 |
| 信号层 | 策略层 | 指标值 | 计算结果，可能失效 |
| 策略层 | 风控层 | 交易指令 | 数量/价格/方向 |
| 风控层 | 执行层 | 合规指令 | 已通过风控 |
| 执行层 | 全链路 | 成交回报 | 状态更新依据 |

## 第三层：单品种处理闭环

一个品种的完整处理周期：

```
T0: K线到达（或定时触发）
    ↓
T1: 更新该品种数据状态（数据层）
    ↓
T2: 引擎层检查并触发处理流程
    ↓
T3: 计算该品种指标（信号层）
    ↓
T4: 生成交易信号
    ↓
T5: 策略层决策（开/平/加/减，数量，价格）
    ↓
T6: 风控层检查（资金够？比例超？模式允许？）
    ↓
T7: 提交订单（或拒绝/缓存）
    ↓
T8: 等待成交回报，更新状态
    ↓
T9: 进入下一周期
```

**关键约束**：
- 同一品种**串行处理**，保证时序一致性
- 不同品种**互不影响**，独立运行
- 每个周期**必须有结果**（成交/拒绝/无信号）

## 第四层：关键业务状态

系统中流转的核心业务状态：

| 状态类型 | 业务含义 | 谁修改 | 谁读取 |
|---------|---------|--------|--------|
| 市场数据 | 最新价格/K线 | 数据层 | 全链路 |
| 指标缓存 | 当前技术指标值 | 信号层 | 策略层 |
| 信号状态 | 当前有效信号 | 策略层 | 风控层 |
| 持仓快照 | 当前持仓数量/成本 | 执行层 | 策略层/风控层 |
| 资金状态 | 可用/冻结/总权益 | 风控层/执行层 | 全链路 |
| 订单状态 | 待报/已报/成交/拒绝 | 执行层 | 策略层 |

**一致性要求**：同一周期内，各层读取的状态必须是同一时刻的快照。

---

## 第五层：关键业务契约

层与层之间的承诺和违约后果：

| 契约点 | 上游承诺 | 下游假设 | 违约后果 |
|--------|---------|---------|---------|
| 数据→指标 | K线完整、时间连续 | 数据可用、格式正确 | 指标计算错误，信号失真 |
| 指标→策略 | 指标实时、计算正确 | 指标可信、不过期 | 决策基于错误信号 |
| 策略→风控 | 指令明确、数量合理 | 指令可执行、不越界 | 风控拒绝，策略失效 |
| 风控→执行 | 指令合规，资金充足 | 指令可提交、不失败 | 订单被拒，机会丧失 |
| 执行→反馈 | 成交及时、回报准确 | 成交可确认、状态可更新 | 持仓混乱，盈亏不清 |

## 第六层：业务边界与容错

异常场景的业务影响和处理原则：

| 异常场景 | 业务影响 | 处理原则 | 回退策略 |
|---------|---------|---------|---------|
| 数据延迟/缺失 | 决策基于旧数据 | 标记数据时效，策略感知 | 暂停交易，等待恢复 |
| 指标计算失败 | 无信号或错误信号 | 不生成信号，不决策 | 保持现状，不操作 |
| 策略信号冲突 | 多空信号同时出现 | 优先级规则 | 按预设规则处理 |
| 风控拒绝 | 指令无法执行 | 记录原因，通知策略 | 调整指令，重新提交 |
| 执行失败 | 订单未成交 | 区分可重试/不可重试 | 重试/放弃/人工介入 |
| 状态不一致 | 持仓/资金对不上 | 停止交易，核对状态 | 人工核对，修正状态 |

## 第七层：Pin策略业务流程（具体品种示例）

以 HOTUSDT 交易对为例，展示完整业务流程：

```
1. 收到新的1分钟K线
   ↓
2. 数据层验证并存储K线
   - 检查时间戳连续性
   - 计算波动率
   ↓
3. 信号层计算指标
   - Z-score（14周期）：当前价格在统计分布中的位置
   - TR比率：波动率相对变化
   - Pine颜色：布林带趋势方向
   ↓
4. 策略层决策（基于Pin状态机）
   
   状态：Initial（初始）
   ├─ 条件：有做空信号 且 Z-score > 阈值
   │   └─ 动作：InitialOpen（初始开空）
   │
   状态：ShortEnter（做空入场）
   ├─ 条件：盈利目标达到 或 止损条件触发
   │   └─ 动作：DoubleClose（双向平仓）
   ├─ 条件：有做多信号 且 价格在对冲区间
   │   └─ 动作：HedgeOpen（对冲开多）
   │
   状态：HedgeEnter（对冲入场）
   ├─ 条件：波动率降低
   │   └─ 动作：ExitHedge（退出对冲）
   │
   状态：PosLocked（仓位锁定）
   └─ 等待总盈亏转正后主仓位平仓
   
   ↓
5. 风控层检查
   - 账户余额是否足够
   - 持仓比例是否超限
   - 是否在交易频率限制内
   ↓
6. 执行层下单
   - 沙盒：模拟成交
   - 实盘：发送到交易所
   ↓
7. 状态更新
   - 记录WAL日志
   - 更新持仓快照
   - 下一周期继续
```

## 第八层：业务权衡与选择

架构决策背后的业务理由：

| 决策 | 选择 | 业务理由 | 代价 | 未来调整 |
|------|------|---------|------|---------|
| 单品种串行 | 保证一致性 | 可预测、易调试、易排查 | 处理速度受限 | 未来可并行多品种 |
| 指标实时计算 | 低延迟 | 信号及时 | 计算资源消耗 | 未来可预计算缓存 |
| 风控前置检查 | 安全第一 | 避免违规损失 | 可能错过机会 | 未来可分级风控 |
| 模拟与实盘同构 | 测试可信 | 沙盒即生产 | 模拟复杂度增加 | 保持 |
| WAL状态恢复 | 数据不丢 | 崩溃可恢复 | 性能损耗 | 保持 |
| 心跳监控 | 运维可见 | 组件存活可知 | 轻微性能开销 | 保持 |
