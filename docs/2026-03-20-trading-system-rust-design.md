================================================================================
一、项目概述
================================================================================

将现有 Go 量化交易系统迁移到 Rust，采用 Barter-rs 风格架构。

核心目标:
- 支持多周期策略并行运行（日线 + 分钟级 + Tick级）
- 高波动时自动切换到高频秒级模式
- 混合持仓模式: 资金池共享，策略持仓独立计算
- 高性能: 1-2 品种秒级/Tick级计算，采用增量计算模式

================================================================================
二、核心需求
================================================================================

1. 多周期并行
   - 日线策略: 低频，单线程足够
   - 分钟策略: 中频，可并行
   - Tick策略: 高频，紧跟市场数据

2. 动态策略切换
   - 根据市场波动率自动启用/禁用策略
   - 高波动品种进入高频计算模式
   - 低波动品种降级到低频模式

3. 混合持仓模式
   - 资金池: 所有策略共享同一个账户资金
   - 持仓结算: 每个策略独立计算持仓，互不干扰
   - 策略间无持仓耦合

4. 高性能要求
   - 1-2 个高波动品种同时处于高频状态
   - 每次 tick 增量计算指标
   - 同步更新，无锁路径追求零延迟

================================================================================
四、各层职责
================================================================================

1. 市场数据层
   - 收集交易所原始数据 (WebSocket)
   - 增量更新 K线/订单簿/Tick
   - 通过 channel 分发数据给策略
   - 技术: Tokio 异步 runtime

2. 指标层
   - 增量计算指标 (EMA, SMA, MACD, 订单簿等)
   - 同步更新，策略直接读取
   - 无锁设计，高频路径零开销

3. 策略层
   - 每个策略独立运行
   - 接收市场数据/指标
   - 产出交易信号
   - 技术: 多线程或 Tokio task

4. 引擎层
   - 资金池: RwLock 保护，低频访问
   - 风控: 统一风控检查，单线程
   - 持仓结算: 策略私有，无锁

================================================================================
五、并发模型
================================================================================

核心设计原则: 高频路径无锁

1. 主事件循环 (Tokio 单线程)
   - 处理所有 tick 数据
   - 增量更新指标
   - 策略信号生成
   - 零锁开销

2. 共享状态访问 (RwLock)
   - 仅用于: 资金池、账户数据
   - 低频访问，不在关键路径

3. 策略并行
   - 每个策略独立 task
   - 通过 channel 接收 tick
   - 策略间完全隔离

4. 指标增量计算
   - EMA: O(1) 更新
   - SMA: O(1) 更新
   - 订单簿: O(log N) 更新
   - K线: O(1) 更新当前K线

================================================================================
七、动态策略切换
================================================================================

1. 模式定义

   enum TradingMode {
       Low,    // 低频模式 (日线)
       Medium, // 中频模式 (分钟)
       High,   // 高频模式 (Tick)
   }

2. 波动率阈值（硬性标准）

   | 策略周期 | 进入高频阈值 | 说明 |
   |---------|-------------|------|
   | 1分钟 | 波动率 >= 3% | K线幅度超过3% |
   | 15分钟 | 波动率 >= 13% | K线幅度超过13% |

   阈值说明:
   - 基于当前K线与前一根K线收盘价的比率计算
   - 达到阈值后策略切换为高频模式
   - 低于阈值时降级为低频模式

3. 高频模式特性

   - 硬性价格判断标准: 必须在价格允许范围内才能交易
   - 指标允许动态延迟: 在价格容许范围内延迟计算
   - 事件驱动: 指标作为事件驱动链的一环

4. 切换机制

   - 监控市场波动率 (基于K线变化率)
   - 波动率超过阈值 → 启用高频策略
   - 波动率低于阈值 → 降级到低频策略
   - 切换时保留仓位状态和指标状态

5. 策略注册表

   struct StrategyRegistry {
       strategies: FnvHashMap<StrategyId, Box<dyn Strategy>>,
       mode: TradingMode,
   }

================================================================================
八、锁与并发控制
================================================================================

1. 核心原则: 简化锁设计

   - 锁外预检所有风控条件，不通过直接返回失败
   - 只有下单时加全局锁
   - 其他操作不加锁，零等待
   - 不需要版本号机制

2. 下单流程

   策略信号产生
       │
       ▼
   ┌───────────────────────────┐
   │ 风控预检（锁外）          │ ← 检查所有条件
   └───────────────────────────┘
       │ 不通过
       ▼ 直接返回失败
       │ 通过
       ▼
   ┌───────────────────────────┐
   │ 加全局锁（下单专用）       │
   └───────────────────────────┘
       │
       ▼
   ┌───────────────────────────┐
   │ 执行下单                  │
   │ 更新持仓/资金             │
   └───────────────────────────┘
       │
       ▼
   ┌───────────────────────────┐
   │ 释放锁                    │
   └───────────────────────────┘

3. 锁的边界

   | 操作 | 是否加锁 | 说明 |
   |------|---------|------|
   | 行情更新/指标计算 | 否 | 高频路径，零锁 |
   | 策略信号生成 | 否 | 读当前状态 |
   | 风控预检 | 否 | 检查所有条件 |
   | 订单执行 | 是 | 全局锁 |
   | 持仓/资金更新 | 是 | 锁内原子更新 |

4. 多线程策略执行

   - 每个策略独立线程/Task
   - 策略间完全隔离，无共享状态
   - 通过 channel 接收市场数据
   - 高频模式下策略专注单品种

九、持仓管理
================================================================================

1. 资金池 (共享)

   struct FundPool {
       total_equity: f64,        // 总权益
       available: f64,           // 可用资金
       positions_value: f64,     // 持仓占用
   }

   访问: RwLock 保护，低频更新

2. 策略持仓 (私有)

   struct StrategyPosition {
       strategy_id: StrategyId,
       symbol: Symbol,
       side: Side,
       qty: f64,
       entry_price: f64,
   }

   每个策略独立计算，不共享锁

3. 结算流程

   订单成交 → 更新策略持仓 → 结算保证金 → 更新资金池

十一、技术选型
================================================================================

| 组件 | 技术选择 | 原因 |
|------|---------|------|
| Runtime | Tokio | 异步 IO，任务调度成熟 |
| 状态管理 | FnvHashMap | O(1) 查找，FnvHash 更快 |
| 共享状态 | RwLock | 读多写少场景优化 |
| 指标存储 | VecDeque | 固定窗口，O(1) 队首队尾 |
| 订单簿 | BTreeMap | 排序，支持范围查询 |
| 序列化 | Serde | Rust 标准 |
| 数值计算 | Rust 原始类型 | 避免 BigDecimal 开销 |

14.2 模块 2: 指标计算层
--------------------------------------------------------------------------------

K线更新触发
    │
    ├─→ ema_12_26.calculate(close_price)      // O(1)
    ├─→ ema_20_50.calculate(close_price)       // O(1)
    └─→ ema_100_200.calculate(close_price)     // O(1)
            │
            ├─→ macd.calculate(ema_fast, ema_slow)
            │       │
            │       ├─→ signal_line.calculate(macd)   // O(1)
            │       └─→ hist = macd - signal           // O(1)
            │
            ├─→ rsi.calculate()                       // O(1)
            │
            ├─→ atr.calculate(high, low, close)      // O(1)
            │
            ├─→ tr_ratio.calculate()
            │       │   // tr_ratio_5d_20d = ATR_5d / ATR_20d
            │       │   // tr_ratio_60min_5h = ATR_60min / ATR_5h
            │
            ├─→ zscore.calculate()
            │       │   // zscore_14_1m = (price - mean_14) / std_14
            │
            └─→ pine_color.detect(macd, signal, hist, rsi)
                    │
                    ├─→ MACD >= Signal && MACD >= 0 → 纯绿
                    ├─→ MACD <= Signal && MACD >= 0 → 淡绿
                    ├─→ MACD <= Signal && MACD <= 0 → 纯红
                    ├─→ MACD >= Signal && MACD <= 0 → 淡红
                    └─→ RSI >= 70 || RSI <= 30 → 紫色

Pine 颜色判断逻辑（基于多周期 EMA 同步）:

| 颜色 | MACD 条件 | RSI 条件 | 说明 |
|------|----------|----------|------|
| 纯绿 | MACD >= Signal && MACD >= 0 | - | 强势多头 |
| 淡绿 | MACD <= Signal && MACD >= 0 | - | 弱势多头 |
| 纯红 | MACD <= Signal && MACD <= 0 | - | 强势空头 |
| 淡红 | MACD >= Signal && MACD <= 0 | - | 弱势空头 |
| 紫色 | - | RSI >= 70 或 RSI <= 30 | 极值区域 |

EMA 周期组:

| 周期组 | 快线 | 慢线 | 应用场景 |
|--------|------|------|---------|
| 12_26 | EMA12 | EMA26 | 1m 周期信号 |
| 20_50 | EMA20 | EMA50 | 辅助信号 |
| 100_200 | EMA100 | EMA200 | 1d 周期信号 |

14.3 模块 3: 策略决策层
--------------------------------------------------------------------------------

每 tick / 每 K线收盘时
    │
    ├─→ VolatilityDetector.check()
    │       │
    │       ├─→ TR_ratio_60min >= 0.03 (3%) → 高波动模式
    │       └─→ TR_ratio_60min < 0.03 → 低波动模式
    │
    ├─→ 高波动模式 → MartinStrategy.check()
    │       │
    │       ├─→ PinStatusDetector.check_long_entry()
    │       │       │
    │       │       ├─→ zscore > 2
    │       │       ├─→ TR_ratio > 1
    │       │       ├─→ pos_norm > 80
    │       │       ├─→ pine_color == 纯绿
    │       │       └─→ 满足 4/7 条件 → LONG_ENTRY
    │       │
    │       ├─→ PinStatusDetector.check_short_entry()
    │       │       │ (类似逻辑，反向)
    │       │
    │       ├─→ PinStatusDetector.check_long_hedge()
    │       │       │
    │       │       ├─→ TR_ratio < 0.15
    │       │       └─→ 满足 4/6 条件 → LONG_HEDGE
    │       │
    │       └─→ PinStatusDetector.check_exit_high_vol()
    │               │
    │               └─→ TR_ratio < 0.15 + 仓位正常 → 退出高波动
    │
    └─→ 低波动模式 → TrendStrategy.check()
            │
            ├─→ TrendStatusDetector.check_long_entry()
            │       │
            │       ├─→ pine_color_all_green()    // 所有周期纯绿
            │       ├─→ TR_ratio_5d_20d > 1
            │       ├─→ ma5_close_position > 70
            │       └─→ 满足条件 → LONG_ENTRY
            │
            ├─→ TrendStatusDetector.check_short_entry()
            │       │ (类似逻辑，反向)
            │
            ├─→ TrendStatusDetector.check_long_exit()
            │       │
            │       ├─→ pine_color_max_period != 纯绿  // 最大周期非纯绿
            │       └─→ 满足条件 → LONG_EXIT
            │
            └─→ TrendStatusDetector.check_short_exit()

TrendStatusDetector 做多条件详情:
- pine 颜色: 全周期纯绿
- TR_ratio: tr_ratio_5d_20d > 1 或 tr_ratio_20d_60d > 1
- MA5 位置: ma5_close_in_all_ma5_pos > 70
- 条件满足: >= 2/3

TrendStatusDetector 做空条件详情:
- pine 颜色: 全周期紫色/纯红
- TR_ratio: 同上
- MA5 位置: ma5_close_in_all_ma5_pos < 30
- 条件满足: >= 2/3

14.4 模块 4: 风控检查层
--------------------------------------------------------------------------------

策略信号产生 (LONG_ENTRY / SHORT_ENTRY / HEDGE / EXIT)
    │
    ├─→ RiskPreChecker.pre_check()     // 锁外执行
    │       │
    │       ├─→ check_account_balance()
    │       │       │ // 可用资金 >= 最低保留
    │       │
    │       ├─→ check_position_ratio()
    │       │       │ // 订单金额 / 总权益 <= 最大比例
    │       │
    │       ├─→ check_symbol_registered()
    │       │       │ // 品种已注册到该策略
    │       │
    │       └─→ check_volatility_mode()
    │               │ // 波动率模式匹配
    │
    │       └─→ 不通过 → 返回 RejectReason
    │
    └─→ 通过 → 进入下单流程

--------------------------------------------------------------------------------

14.5 模块 5: 下单执行层
--------------------------------------------------------------------------------

PreCheck 通过
    │
    ├─→ GlobalLock.acquire()          // 获取全局锁
    │       └─→ 失败 → 返回 LockFailed
    │
    ├─→ RiskReChecker.re_check()      // 锁内再次风控 (可选)
    │
    ├─→ OrderService.place_order()
    │       │
    │       ├─→ 市价单
    │       │       └─→ gateway.place_market_order()
    │       │
    │       └─→ 限价单
    │               └─→ gateway.place_limit_order()
    │
    ├─→ 等待成交回报
    │       │
    │       └─→ OrderResult { status: FILLED / REJECTED / CANCELLED }
    │
    └─→ GlobalLock.release()           // 释放全局锁

说明:
- 限价单挂单时不冻结保证金，成交后才占用
- 不需要预占用机制
- 下单失败不重试，直接返回

--------------------------------------------------------------------------------

14.6 模块 6: 持仓 / 资金更新层
--------------------------------------------------------------------------------

成交回报到达
    │
    ├─→ PositionManager.update_local()
    │       │
    │       ├─→ 开多: LongQty += qty, LongAvgPrice 更新
    │       ├─→ 开空: ShortQty += qty, ShortAvgPrice 更新
    │       ├─→ 平多: LongQty -= qty, 计算已实现盈亏
    │       └─→ 平空: ShortQty -= qty, 计算已实现盈亏
    │
    ├─→ FundPool.update()
    │       │
    │       ├─→ 盈利时: CurrentBalance += pnl
    │       └─→ 亏损时: CurrentBalance -= pnl
    │
    ├─→ StrategyPosition.update()      // 策略私有持仓
    │       │
    │       └─→ 独立计算该策略的持仓和盈亏
    │
    └─→ PersistenceService.save()
            │
            ├─→ SQLite: 交易记录、持仓快照
            └─→ /dev/shm: K线备份、缓存

14.7 模块 7: 策略状态机
--------------------------------------------------------------------------------

马丁格尔策略状态流转:

    ┌─────────────────────────────────────────────────────────────┐
    │                         IDLE                                │
    │                    (等待入场)                               │
    └─────────────────────────┬───────────────────────────────────┘
                              │
                              │ check_long_entry() 或 check_short_entry()
                              ▼
    ┌─────────────────────────────────────────────────────────────┐
    │                        OPENING                              │
    │                   (马丁格尔加仓中)                           │
    └───────┬─────────────────────┬───────────────────────────────┘
            │                     │
            │ check_long_hedge() │ check_long_exit()
            ▼                     ▼
    ┌───────────────┐     ┌───────────────┐
    │   HEDGING      │     │   CLOSING     │
    │   (对冲中)     │     │   (平仓中)    │
    └───────┬───────┘     └───────┬───────┘
            │                     │
            │ 对冲盈利回落 1%      │ 持仓清零
            ▼                     ▼
    ┌───────────────┐     ┌───────────────┐     check_exit_high_vol()
    │  CLOSING_ALL  │     │     IDLE      │◄────────────────────┐
    │   (全平)       │     │               │                      │
    └───────┬───────┘     └───────────────┘                      │
            │                                                       │
            │ 趋势确认                                              │
            └──────────────────────┐                                │
                                   │                                │
                                   ▼                                │
                          ┌─────────────────┐                      │
                          │   TREND_MODE    │──────────────────────┘
                          │  (趋势跟踪模式)   │    趋势结束
                          └─────────────────┘

状态说明:

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| IDLE | 等待入场 | 无持仓，等待信号 |
| OPENING | 马丁加仓中 | 入场后，每次加仓条件满足 |
| HEDGING | 对冲中 | 多单回落时做空对冲 |
| CLOSING | 平仓中 | 达到平仓条件 |
| CLOSING_ALL | 全平 | 对冲盈利回落 1% |
| TREND_MODE | 趋势跟踪 | 高波动结束，进入趋势 |

15.4 架构共同思想
--------------------------------------------------------------------------------

从多个参考项目中提炼的共识架构模式:

1. **事件驱动** - 所有交易系统都采用事件驱动架构
2. **无锁高频路径** - tick 处理无锁是高性能共识
3. **Trait 抽象** - 可插拔组件是标准设计模式
4. **Actor 风格** - 状态封装 + 消息传递

15.5 需要避免的问题
--------------------------------------------------------------------------------

参考项目中发现的常见问题:

| 问题 | 来源 | 后果 | 我们的方案 |
|------|------|------|-----------|
| panic!() 错误处理 | barter-rs | 解析错误导致崩溃 | 全部使用 Result |

| 测试不足 | barter-rs (1个集成测试) | 重构风险高 | 全面测试覆盖 |
