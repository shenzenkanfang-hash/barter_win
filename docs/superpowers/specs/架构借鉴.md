一、核心架构模式
================================================================================

1.1 Processor Trait 模式
--------------------------------------------------------------------------------
事件处理的通用抽象，所有核心组件都实现此 trait:

```rust
pub trait Processor<E> {
    type Audit;
    fn process(&mut self, event: E) -> Self::Audit;
}
```

好处:
- 统一事件处理接口
- 便于测试和 mock
- 支持组合和嵌套

1.2 Engine 泛型模式
--------------------------------------------------------------------------------
核心事件处理器，使用泛型支持灵活扩展:

```rust
pub struct Engine<Clock, State, Strategy, Risk> {
    clock: Clock,
    state: State,
    strategy: Strategy,
    risk_manager: Risk,
}
```

好处:
- 策略和风控逻辑可插拔
- 支持回测和实盘不同实现
- 单一入口简化架构

1.3 Trait 抽象体系
--------------------------------------------------------------------------------
关键 traits:

| Trait | 用途 | 关键方法 |
|-------|------|---------|
| AlgoStrategy | 算法策略 | generate_algo_orders() |
| ClosePositionsStrategy | 平仓策略 | generate_close_orders() |
| OnDisconnectStrategy | 断开处理 | on_disconnect() |
| RiskManager | 风控管理 | check_orders() |
| ExecutionClient | 执行客户端 | submit_order(), cancel_order() |
| MarketStream | 市场数据流 | init() |

二、模块化设计
================================================================================

2.1 Workspace 多 Crate 结构
--------------------------------------------------------------------------------
barter-rs 采用 workspace 模式，清晰分层:

barter/           - 核心引擎
barter-data/      - 市场数据 (WebSocket)
barter-execution/ - 订单执行
barter-integration/ - 底层协议框架
barter-instrument/ - 合约/资产数据结构
barter-macro/     - 过程宏

借鉴: 我们采用类似的分层:

market/      - 市场数据层
indicator/   - 指标层
strategy/    - 策略层
engine/      - 引擎层
account/     - 账户层

2.2 目录命名规范
--------------------------------------------------------------------------------
- 模块目录: snake_case (engine/state/)
- 单一模块: mod.rs 导出公共项
- 功能文件: 多个相关项放一个文件

4.1 派生宏顺序
--------------------------------------------------------------------------------
```rust
#[derive(
    Debug,           // 公共类型始终派生
    Clone,           // 用于所有权上下文
    Eq,              // 用于有 PartialEq 的类型
    PartialEq,        // 相等比较
    Ord,             // 用于有 PartialOrd 的类型
    PartialOrd,       // 排序
    Hash,             // 用于集合
    Deserialize,      // Serde
    Serialize,        // Serde
    Error,            // thiserror
)]
```

4.2 错误类型模式
--------------------------------------------------------------------------------
```rust
#[derive(Debug, Clone, Eq, PartialEq, Error)]
pub enum MyError {
    #[error("IndexError: {0}")]
    IndexError(#[from] IndexError),

    #[error("ExecutionError: {0}")]
    ExecutionError(String),

    #[error("market data: {0}")]
    MarketData(#[from] DataError),
}
```

原则:
- 使用 #[from] 自动转换
- 错误消息包含上下文
- 实现 Debug, Clone, Deserialize, Serialize

五、数据流设计
================================================================================

5.1 实盘交易流程
--------------------------------------------------------------------------------
WebSocket 接收 tick
    │
    ├─> MarketData 更新
    │
    ├─> 指标增量计算 (O(1))
    │
    ├─> 策略信号生成
    │
    ├─> 风控预检 (锁外)
    │
    ├─> GlobalLock.acquire()
    │
    ├─> 订单执行
    │
    └─> GlobalLock.release()

5.2 Engine 事件处理序列
--------------------------------------------------------------------------------
Engine::process(event)
    │
    ├─> update_state()       // 更新引擎状态
    │
    ├─> generate_orders()    // 策略生成订单
    │       │
    │       └─> RiskManager::check()  // 风控审查
    │
    ├─> execute_orders()     // 执行订单
    │
    └─> generate_audit()     // 生成审计记录

六、需要避免的坑
================================================================================

6.1 panic!() 错误处理
--------------------------------------------------------------------------------
barter-rs 问题: 大量使用 panic!() 而非 Result

我们必须:
- 所有解析逻辑返回 Result
- 市场数据错误不能导致程序崩溃
- 交易所 API 变更要有容错

6.2 高频路径锁竞争
--------------------------------------------------------------------------------
barter-rs 问题: RwLock 在高频路径造成竞争

我们必须:
- tick 处理路径无锁
- 指标增量计算无锁
- 锁仅用于低频操作 (下单/资金更新)

6.3 过多 clone()
--------------------------------------------------------------------------------
barter-rs 问题: 77 个文件 344 处 clone

我们必须:
- 优先使用引用 &T
- 使用 Arc/Rc 共享数据
- 避免不必要的数据复制

6.4 测试覆盖不足
--------------------------------------------------------------------------------
barter-rs 问题: workspace 仅 1 个集成测试

我们必须:
- 核心 Engine 有单元测试
- 策略逻辑有单元测试
- 指标计算有单元测试
- 关键路径有集成测试

七、我们的架构优势
================================================================================

相比 barter-rs，我们的架构:

| 方面 | barter-rs | 我们的设计 |
|------|----------|-----------|
| 策略分层 | 单一 Engine | 日线+分钟级+Tick 三层 |
| 指标复杂度 | EMA/MACD/RSI/KDJ 等 | 核心 3 指标 (TR/Pine/价格位置) |
| 持仓模式 | 单一持仓 | 资金池共享，策略持仓私有 |
| 模式切换 | 无 | 动态切换高/低频模式 |
| 风控检查 | RiskManager trait | 锁外预检 + 锁内执行 |
