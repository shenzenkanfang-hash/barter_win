一、项目概述
================================================================================

将现有 Go 量化交易系统迁移到 Rust，采用 Barter-rs 风格架构。

核心目标:
- 支持多周期策略并行运行（日线 + 分钟级 + Tick级）
- 高波动时自动切换到高频秒级模式
- 混合持仓模式: 资金池共享，策略持仓独立计算
- 高性能: 1-2 品种秒级/Tick级计算，采用增量计算模式

================================================================================
二、核心需求
================================================================================

1. 多周期并行
   - 日线策略: 低频，单线程足够
   - 分钟策略: 中频，可并行
   - Tick策略: 高频，紧跟市场数据

2. 动态策略切换
   - 根据市场波动率自动启用/禁用策略
   - 高波动品种进入高频计算模式
   - 低波动品种降级到低频模式

3. 混合持仓模式
   - 资金池: 所有策略共享同一个账户资金
   - 持仓结算: 每个策略独立计算持仓，互不干扰
   - 策略间无持仓耦合

4. 高性能要求
   - 1-2 个高波动品种同时处于高频状态
   - 每次 tick 增量计算指标
   - 同步更新，无锁路径追求零延迟

================================================================================
四、各层职责
================================================================================

1. 市场数据层
   - 收集交易所原始数据 (WebSocket)
   - 增量更新 K线/订单簿/Tick
   - 通过 channel 分发数据给策略
   - 技术: Tokio 异步 runtime

2. 指标层
   - 增量计算指标 (EMA, SMA, MACD, 订单簿等)
   - 同步更新，策略直接读取
   - 无锁设计，高频路径零开销

3. 策略层
   - 每个策略独立运行
   - 接收市场数据/指标
   - 产出交易信号
   - 技术: 多线程或 Tokio task

4. 引擎层
   - 资金池: RwLock 保护，低频访问
   - 风控: 统一风控检查，单线程
   - 持仓结算: 策略私有，无锁

================================================================================
五、并发模型
================================================================================

核心设计原则: 高频路径无锁

1. 主事件循环 (Tokio 单线程)
   - 处理所有 tick 数据
   - 增量更新指标
   - 策略信号生成
   - 零锁开销

2. 共享状态访问 (RwLock)
   - 仅用于: 资金池、账户数据
   - 低频访问，不在关键路径

3. 策略并行
   - 每个策略独立 task
   - 通过 channel 接收 tick
   - 策略间完全隔离

4. 指标增量计算
   - EMA: O(1) 更新
   - SMA: O(1) 更新
   - 订单簿: O(log N) 更新
   - K线: O(1) 更新当前K线

================================================================================
六、指标设计
================================================================================

1. 核心特性: 事件驱动 + 单向线性步进

   - 事件驱动: 指标是事件驱动链的一环，tick 触发增量更新
   - 单向线性步进: 指标状态只增不减，历史窗口平移
   - 零锁设计: 增量计算在主事件循环同步完成

2. 支持增量计算的指标

| 指标 | 更新复杂度 | 说明 |
|------|-----------|------|
| EMA | O(1) | 指数移动平均，天然递归 |
| SMA | O(1) | 滑动窗口，维护队列 |
| MACD | O(1) | 基于 EMA 的组合 |
| KDJ | O(1) | 基于 SMA/EMA |
| 布林带 | O(1) | 基于 SMA + 标准差 |
| 订单簿 | O(log N) | BTreeMap 维护档位 |
| K线 | O(1) | 只更新当前K线 |

3. 指标更新流程

   tick 到达
       │
       ▼
   ┌─────────────────┐
   │ 增量更新指标    │  ← 同步执行，无锁
   └─────────────────┘
       │
       ▼
   ┌─────────────────┐
   │ 策略读取指标    │  ← 直接访问共享数据
   └─────────────────┘
       │
       ▼
   ┌─────────────────┐
   │ 生成交易信号    │
   └─────────────────┘

4. 高频模式下的指标特性

   - 价格硬限制: 必须满足价格条件才能触发交易
   - 指标允许延迟: 在价格容许范围内延迟计算
   - 降级策略: 价格超出范围时跳过该 tick 的指标更新

5. 指标共享方式
   - 基础指标: 策略共享同一实例 (减少计算)
   - 复合指标: 可配置共享或独立

================================================================================
七、动态策略切换
================================================================================

1. 模式定义

   enum TradingMode {
       Low,    // 低频模式 (日线)
       Medium, // 中频模式 (分钟)
       High,   // 高频模式 (Tick)
   }

2. 波动率阈值（硬性标准）

   | 策略周期 | 进入高频阈值 | 说明 |
   |---------|-------------|------|
   | 1分钟 | 波动率 >= 3% | K线幅度超过3% |
   | 15分钟 | 波动率 >= 13% | K线幅度超过13% |

   阈值说明:
   - 基于当前K线与前一根K线收盘价的比率计算
   - 达到阈值后策略切换为高频模式
   - 低于阈值时降级为低频模式

3. 高频模式特性

   - 硬性价格判断标准: 必须在价格允许范围内才能交易
   - 指标允许动态延迟: 在价格容许范围内延迟计算
   - 事件驱动: 指标作为事件驱动链的一环

4. 切换机制

   - 监控市场波动率 (基于K线变化率)
   - 波动率超过阈值 → 启用高频策略
   - 波动率低于阈值 → 降级到低频策略
   - 切换时保留仓位状态和指标状态

5. 策略注册表

   struct StrategyRegistry {
       strategies: FnvHashMap<StrategyId, Box<dyn Strategy>>,
       mode: TradingMode,
   }

================================================================================
八、锁与并发控制
================================================================================

1. 核心原则: 乐观并发

   - 先检查风控，不通过直接返回失败
   - 通过后才加锁复核 + 下单
   - 其他操作不加锁，零等待

2. 版本号机制

   struct VersionedState {
       version: u64,           // 版本号，只有锁内能修改
       positions: Positions,    // 本地持仓快照
       funds: Funds,           // 本地资金快照
   }

   流程:
   1. 检查风控 → 不通过直接返回失败
   2. 获取当前版本号
   3. 加锁
   4. 核对版本号是否一致
   5. 再次复核风控
   6. 执行下单
   7. 更新持仓和资金
   8. 版本号+1
   9. 释放锁

3. 锁的边界

   | 操作 | 是否加锁 | 说明 |
   |------|---------|------|
   | 行情更新/指标计算 | 否 | 高频路径，零锁 |
   | 策略信号生成 | 否 | 读当前状态 |
   | 风控预检 | 否 | 不通过直接失败 |
   | 订单执行 | 是 | 仅在确认后加锁 |
   | 持仓/资金更新 | 是 | 锁内原子更新 |

4. 多线程策略执行

   - 每个策略独立线程/Task
   - 策略间完全隔离，无共享状态
   - 通过 channel 接收市场数据

九、持仓管理
================================================================================

1. 资金池 (共享)

   struct FundPool {
       total_equity: f64,        // 总权益
       available: f64,           // 可用资金
       positions_value: f64,     // 持仓占用
   }

   访问: RwLock 保护，低频更新

2. 策略持仓 (私有)

   struct StrategyPosition {
       strategy_id: StrategyId,
       symbol: Symbol,
       side: Side,
       qty: f64,
       entry_price: f64,
   }

   每个策略独立计算，不共享锁

3. 结算流程

   订单成交 → 更新策略持仓 → 结算保证金 → 更新资金池

十一、技术选型
================================================================================

| 组件 | 技术选择 | 原因 |
|------|---------|------|
| Runtime | Tokio | 异步 IO，任务调度成熟 |
| 状态管理 | FnvHashMap | O(1) 查找，FnvHash 更快 |
| 共享状态 | RwLock | 读多写少场景优化 |
| 指标存储 | VecDeque | 固定窗口，O(1) 队首队尾 |
| 订单簿 | BTreeMap | 排序，支持范围查询 |
| 序列化 | Serde | Rust 标准 |
| 数值计算 | Rust 原始类型 | 避免 BigDecimal 开销 |

十二、性能目标
================================================================================

| 场景 | 目标延迟 | 实现方式 |
|------|---------|---------|
| Tick 接收 | < 1ms | Tokio 异步 |
| 指标更新 | < 0.1ms | 增量计算 O(1) |
| 策略判断 | < 0.1ms | 单线程同步 |
| 信号输出 | < 1ms | Channel 传递 |
| 总延迟 | ~5-10ms | 满足秒级交易 |

十三、对比现有 Go 项目
================================================================================

| 对比项 | Go 微服务 | Rust 单体 |
|--------|---------|---------|
| 架构 | 多进程 gRPC | 单进程 trait 调用 |

| 维护 | 多服务协调难 | 单一代码库 |
