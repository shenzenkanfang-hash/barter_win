## 1. f_traits 独立 crate 评估

### 1.1 当前状态
- 接口定义在 `f_engine/src/interfaces/`
- 存在潜在循环依赖问题（f_engine 定义接口，下层实现）

### 1.2 评估方案

#### 方案 A：将 interfaces 移至 a_common（推荐）
```
优点：
- a_common 是权威基础层，所有 crate 都可访问
- 消除循环依赖
- 统一接口定义位置

缺点：
- a_common 需要依赖更多类型
- 可能需要拆分 a_common 为更细粒度的子模块

实施步骤：
1. 将 f_engine/src/interfaces/ 移至 a_common/src/interfaces/
2. 更新 f_engine 的 re-exports
3. 更新所有实现者的 import
```

#### 方案 B：新建 f_traits crate
```
crate 结构：
f_engine/
  ├── f_traits/           # 新建独立接口 crate
  ├── f_engine/           # 核心引擎
  └── f_order/            # 订单模块

优点：
- 完全解耦
- 接口可独立演进

缺点：
- 增加 crate 数量
- 需要更复杂的 workspace 配置
```

### 1.3 评估结论
**推荐方案 A**：将 interfaces 移至 a_common
- 改动范围可控
- 符合当前六层架构
- 实施成本低

## 2. 异步 Trait 标准化评估

### 2.1 当前状态
- 部分 Trait 使用同步方法
- 部分使用 `#[async_trait]`
- 异步边界不统一

### 2.2 标准化方案

#### 规范要求
```
所有异步接口统一使用 #[async_trait] + Send + Sync

示例：
#[async_trait]
pub trait MarketDataProvider: Send + Sync {
    async fn next_tick(&self) -> Option<MarketTick>;
    async fn get_klines(&self, symbol: &str, period: &str) -> Vec<MarketKLine>;
}

同步接口示例：
pub trait RiskChecker: Send + Sync {
    fn pre_check(&self, request: &OrderRequest) -> RiskCheckResult;
}
```

### 2.3 评估结论
**实施建议**：
- 异步方法使用 `#[async_trait]`
- 同步方法直接定义
- 统一 `Send + Sync` 约束
- 主运行时使用 tokio

## 3. 事件驱动架构评估

### 3.1 当前状态
- 同步调用模式
- 阻塞式检查
- 缺乏事件总线

### 3.2 事件驱动方案

#### 方案 A：Channel-based Event Bus（推荐）
```
使用 tokio channel 实现事件总线

事件类型：
enum EngineEvent {
    MarketData(MarketTick),
    OrderFilled(OrderResult),
    RiskAlert(RiskWarning),
    PositionUpdate(Position),
}

优点：
- 简单直接
- 与 tokio 生态集成
- 低开销

缺点：
- 需要手动管理订阅
```

#### 方案 B：Publish-Subscribe Pattern
```
使用 crate: tokio::sync::broadcast

优点：
- 多订阅者支持
- 解耦更彻底

缺点：
- 复杂度增加
- 需要事件路由
```

#### 方案 C：Actor Pattern
```
使用 crate: actix 或 xtract

优点：
- 状态封装
- 并发安全

缺点：
- 学习成本高
- 过度设计风险
```

### 3.3 评估结论
**推荐方案 A**：Channel-based Event Bus
- 适合当前系统规模
- 与现有架构兼容
- 实施成本低
