## Claude Code Rust 项目准则 v1.0

---

### 一、交互原则

**1.1 上下文优先**
- 每次对话开头主动提供项目结构概览
- 涉及修改时，先说明影响范围（crate/模块/文件）
- 跨 crate 改动必须显式列出依赖关系变更

**1.2 渐进式确认**
- 架构级改动：先出方案，确认后再实现
- 破坏性变更：列出 breaking changes 清单
- 性能敏感代码：提供 benchmark 对比说明

**1.3 术语一致性**
- 使用标准 Rust 术语（crate/module/trait/impl）
- 避免混用其他语言概念（如 "类"、"接口"）
- 异步相关统一说 async/await，不混用 "协程"

---

### 二、代码生成规范

**2.1 风格约定**
- 遵循 `cargo fmt` 默认配置
- 优先标准库，其次生态主流（tokio/serde/thiserror）
- 错误处理：显式区分 `Result` 与 `Option`，不用 `unwrap` 生产代码
- 文档注释：pub 项必须有 `///`，含 Examples 段落

**2.2 工程结构**
- 多 crate 项目用 workspace，单 crate 不用虚拟目录
- 二进制与库分离：`src/lib.rs` + `src/main.rs` 或 `src/bin/`
- 测试分层：单元测试放文件底部，集成测试放 `tests/`，doc 测试写进注释

**2.3 依赖管理**
- 加依赖前确认是否已存在功能重复项
- 版本约束用 caret (`^`)，不用通配符
- 新增依赖需说明选型理由（性能/维护/功能）

---

### 三、文档生成标准

**3.1 架构文档**
- 位置：`docs/arch/` 或项目根目录 `ARCHITECTURE.md`
- 必须包含：数据流图、模块边界、错误传播路径
- 更新触发条件：新增 crate、修改核心 trait、变更并发模型

**3.2 API 文档**
- 由 `cargo doc` 生成，不手写重复
- 补充内容：设计决策（Why）、性能特征（Complexity）、线程安全（Send/Sync）
- 复杂类型附使用示例，跨 await 点注明取消安全性

**3.3 变更日志**
- 格式：Keep a Changelog
- 版本号：SemVer，alpha/beta 标 pre-release
- 条目分类：Added/Changed/Deprecated/Removed/Fixed/Security

**3.4 运行手册**
- 位置：`docs/runbook/` 或 `RUNBOOK.md`
- 包含：环境变量清单、关键指标说明、故障排查步骤
- 与代码同步更新，版本号与发布标签一致

---

### 四、审查清单

**4.1 生成后自检**
- [ ] `cargo check` 无警告（允许预期内的 deprecated）
- [ ] `cargo clippy` 全通过（复杂规则可显式允许并注释原因）
- [ ] `cargo test` 全绿（包括 doc test）
- [ ] `cargo doc` 无 broken link

**4.2 交付前确认**
- [ ] 新增 pub 项有文档 + 示例
- [ ] 错误类型实现 `std::error::Error` 且可打印
- [ ] 异步边界有 `Send + 'static` 约束说明
- [ ] unsafe 代码块有 SAFETY 注释

---

### 五、禁止行为

- 不生成未请求的二进制文件或脚本
- 不假设外部工具存在（先询问或提供安装指引）
- 不混用同步/异步代码而不加桥接说明
- 不隐藏编译器警告（要么修复，要么显式允许并解释）

## Claude Code Rust 量化交易准则 v1.0

---

### 一、领域术语规范

| 场景 | 标准术语 | 禁用表述 |
|------|---------|---------|
| 市场数据 | tick/kline/book/quote | "行情"、"数据包" |
| 订单流 | order/fill/position/PnL | "成交回报"、"盈亏" |
| 策略逻辑 | alpha/signal/risk check | "策略计算"、"风控判断" |
| 执行路径 | hot path/warm path/cold path | "主流程"、"后台任务" |
| 延迟分级 | data latency/processing latency/total latency | "延迟1"、"延迟2" |

---

### 二、模块边界原则

**2.1 数据层（b_data_*/c_data_process）**
- 原始数据不可变，转换产生新类型
- 时间戳必须带时区（`DateTime<Utc>`），禁止裸 `i64`
- 序列号连续性强校验，缺口即告警

**2.2 策略层（d_strategy_*/e_signal_*）**
- 纯函数核心：输入（market state）→ 输出（signal vector）
- 状态隔离：策略实例 per 交易对，禁止全局可变状态
- 回测/实盘统一接口，差异仅数据源实现

**2.3 执行层（f_execution_*/g_order_mgmt）**
- 订单状态机显式定义：Created → Sent → Acked → Partial → Filled / Rejected / Cancelled
- 所有状态转移落日志，支持重放审计
- 拒单/超时自动重试策略可配置（指数退避）

**2.4 风控层（h_risk_*）**
- 检查点前置：报单前预检、成交后校验、定时全量扫描
- 硬限制（kill switch）与软限制（告警）分层
- 风控规则热更新，不重启服务

---

### 三、性能工程规范

**3.1 延迟预算分配**
```
总延迟 < 10ms (高频) / < 100ms (中频) / < 1s (低频)

分解示例（高频）：
- 网络 I/O: 2-3ms  ← 不可控，监控即可
- 数据解码: 0.5ms  ← 用 zero-copy parser（simd-json 等）
- 策略计算: 1-2ms  ← 预分配、无锁、批处理
- 风控检查: 0.5ms  ← 规则编译为决策树或 LUT
- 订单构造: 0.3ms  ← 对象池复用
- 网络发送: 2-3ms  ← 批量发送、TCP_NODELAY 调优
```

**3.2 内存管理**
- 热路径禁止分配：用 `ringbuf`、`crossbeam` 预分配队列
- K 线数据用 arena allocator 或 mmap 文件
- 大对象（历史数据）走 cold path，异步加载

**3.3 并发模型**
- 数据流：单生产者多消费者（SPMC），每交易对独立线程/任务
- 共享状态：用 `dashmap` 或分片 `RwLock`，禁止全局 `Mutex`
- 异步边界：明确标注 `Send + 'static`，跨线程传递用消息而非 Arc

---

### 四、数据完整性铁律

**4.1 时间序列连续性**
- K 线序列号单调递增，检测到缺口立即标记数据质量降级
- 交易所重连后补全缺失数据，补不到则策略暂停（不盲跑）

**4.2 订单-成交对账**
- 内部订单簿 vs 交易所回报 实时核对
- 差异超过阈值（如 1 秒无匹配成交）自动熔断
- 日终对账报告生成，人工确认后方可次日开盘

**4.3 精度处理**
- 价格用 `Decimal`（rust-decimal），禁止 `f64` 金融计算
- 数量精度按交易所 instrument 定义，舍入模式显式配置（HALF_UP 等）

---

### 五、故障与恢复

**5.1 分级熔断**

| 级别 | 触发条件 | 动作 | 恢复方式 |
|------|---------|------|---------|
| L1 策略 | 单笔信号异常 | 跳过该信号，记日志 | 自动 |
| L2 账户 | 连续 N 单拒单/超时 | 暂停该账户交易 | 人工确认 |
| L3 市场 | 数据中断 > T 秒 | 全策略空仓，撤所有单 | 人工 + 数据恢复 |
| L4 系统 | 内存/CPU 临界 | 优雅退出，持久化状态 | 重启 + 状态回放 |

**5.2 状态持久化**
- 关键状态（持仓、未完成订单）写 WAL，本地 + 远程双副本
- 重启后从 WAL 回放，恢复到崩溃前一致状态
- 回放速度 > 实时 10 倍，确保快速恢复

---

### 六、监控与可观测

**6.1 指标分级**
```
L1 业务指标（告警用）：
- 盈亏、夏普、最大回撤
- 滑点分布（实际成交 vs 信号价格）
- 订单成交率、取消率

L2 系统指标（调试用）：
- 各组件延迟 P50/P99/P999
- 队列深度、内存分配速率
- GC/调度延迟（tokio 指标）

L3 诊断指标（排查用）：
- 逐笔延迟追踪（trace ID 串联）
- 数据缺口详细日志
- 风控规则命中统计
```

**6.2 日志规范**
- 结构化 JSON，字段统一：`ts`, `level`, `component`, `trace_id`, `market`, `msg`
- 热路径采样日志（1/1000），全量写内存 ring buffer 按需导出
- 禁止日志反压：异步 channel + 丢弃策略（优先保延迟）

---

### 七、回测与实盘一致性

**7.1 数据源抽象**
- 统一接口：`DataSource`，实现 `LiveSource` 和 `ReplaySource`
- 回测用历史数据文件（parquet/zstd 压缩），按时间戳流式回放
- 模拟撮合：支持逐笔回放的精细撮合 vs 简化撮合（K 线级）可选

**7.2 延迟模拟**
- 回测注入网络延迟分布（基于历史实测或假设分布）
- 订单成交概率按滑点模型模拟，禁止 "假设全部成交"

**7.3 确定性保证**
- 同一份代码 + 同一份数据 + 同一份配置 = 可复现结果
- 随机数种子固定，多线程执行顺序记录并可重放

---

### 八、安全与合规

**8.1 密钥管理**
- API Key 不走代码仓库，运行时从 Vault/AWS KMS 拉取
- 内存中敏感数据用 `secrecy` crate 包裹，禁止 `Debug` 泄露
- 日志脱敏：Key 只打前 4 后 4，中间掩码

**8.2 操作审计**
- 人工指令（手动撤单、参数修改）全记录：who/when/what/before/after
- 策略参数变更需双人复核，Git 提交关联工单号

### 十、市场数据管道专项

**10.1 多交易所聚合**
- 统一 symbol 命名空间：内部用 `Exchange:BaseQuote`（如 `BINANCE:BTCUSDT`），映射层隔离交易所差异
- 时间对齐：不同交易所时钟漂移监控，超过 100ms 标记套利信号不可信
- 优先级策略：主数据源失效时，备用源切换延迟 < 500ms，切换事件全链路追踪

**10.2 数据质量标记**
```rust
// 每个 tick/kline 携带质量元数据
pub struct DataQuality {
    pub source_status: SourceStatus,      // Live / Delayed / Stale / Backfill
    pub sequence_gap_detected: bool,
    pub exchange_timestamp_drift_ms: i64, // 负值表示交易所时钟落后
    pub local_arrival_latency_us: u64,
}
```
- 策略消费时显式处理质量降级：忽略、加权、或暂停交易

**10.3 快照与增量**
- 订单簿维护：初始 snapshot + 后续 diff，校验 checksum
- diff 应用失败（如序列号不连续）即触发全量重同步
- 重同步期间该 symbol 交易暂停，避免 stale 数据驱动决策

### 十一、策略执行引擎

**11.1 信号生命周期**
```
Raw Signal (策略生成)
    ↓
Validity Check (时效性：信号基于的数据是否过期)
    ↓
Risk Filter (仓位、敞口、频率限制)
    ↓
Execution Convert (信号 → 具体订单参数：价格、数量、类型)
    ↓
Pre-trade Check (最终校验：余额、风控规则)
    ↓
Dispatch (发往交易所)
```
- 每阶段耗时独立监控，瓶颈定位到具体环节

**11.2 订单类型策略**
| 场景 | 首选类型 | 备选方案 | 禁止行为 |
|------|---------|---------|---------|
| 被动做市 | Post-only Limit | — | 市价单吃单 |
| 主动吃单 | IOC Limit | FOK | 无保护市价单 |
| 紧急平仓 | 市价带滑点上限 | TWAP 拆分 | 裸市价单 |
| 大额拆单 | VWAP/TWAP | 冰山订单 | 单笔大单冲击市场 |

**11.3 信号冲突解决**
- 同 symbol 多策略：优先级权重或投票机制，禁止同时多空对锁
- 信号撤销：策略生成 cancel 指令，Execution Engine 追踪原订单状态
- 部分成交处理：剩余 quantity 重评估，可能改价/撤单/保留

---

### 十二、仓位与盈亏核算

**12.1 仓位模型**
- 支持多会计方法：FIFO / LIFO / 平均成本，方法切换需日终清盘
- 未实现盈亏按标记价格实时计算，已实现盈亏按成交确认
- 跨交易所仓位聚合，总敞口监控防止超限额

**12.2 对账机制**
| 对账类型 | 频率 | 差异处理 |
|---------|------|---------|
| 内部 vs 交易所 | 每笔成交后 | 差异 > 0 立即告警，人工介入 |
| 策略预期 vs 实际成交 | 每笔 | 滑点分析，超阈值记异常 |
| 日终总账 | 每日收盘后 | 生成对账报告，差异追踪至 root cause |

**12.3 资金费率与资金成本**
- 永续合约资金费率自动抓取，持仓成本实时更新
- 借贷利率（杠杆场景）纳入盈亏计算，策略信号考虑 carry cost

### 十三、风险管理深化

**13.1 实时风险指标**
```
希腊字母监控（期权策略）：
- Delta 敞口：按 underlying 聚合，对冲比率自动计算
- Gamma / Theta / Vega：突破阈值限制新仓

期货/现货：
- 保证金利用率：逐仓/全仓分别监控
- 爆仓价格距离：按 mark price 实时计算
- 相关性矩阵：持仓品种间相关性突变检测（如历史 0.1 → 当前 0.8）
```

**13.2 压力测试**
- 每日盘前：历史极端行情回放（如 312、519 暴跌）
- 场景参数：价格跳变 10%、波动率翻倍、流动性归零
- 输出：最大回撤、保证金缺口、强平顺序

**13.3  kill switch 分级**
| 触发器 | 延迟要求 | 动作 | 人工覆盖 |
|--------|---------|------|---------|
| 单笔亏损 > X | < 10ms | 撤全仓、暂停策略 | 需双人确认恢复 |
| 分钟级回撤 > Y% | < 100ms | 减仓至 1/10 | 可一键恢复 |
| 交易所 API 异常 | < 1s | 切换备用账户 | 自动 |
| 系统资源临界 | < 5s | 优雅退出 | 需重启 |

### 十四、执行算法（Execution Algorithms）

**14.1 TWAP 实现要点**
- 时间窗口切分：均匀或按成交量预测分布
- 紧急完成度：剩余时间 < 20% 时加速，剩余量 > 30% 时改 IOC
- 参与率限制：每分钟成交量不超过市场总量的 5%

**14.2 智能订单路由（SOR）**
- 多交易所比价：考虑价格、深度、手续费、提现成本
- 拆分执行：大单按流动性比例拆至各 venue
- 路由决策延迟 < 2ms，否则退化为单交易所

**14.3 自适应滑点模型**
```rust
// 基于实时市场状态动态调整
pub fn adaptive_slippage(
    order_size: Decimal,
    book_depth: &OrderBook,
    recent_volatility: f64,
    urgency: UrgencyLevel,
) -> SlippageEstimate;
```
- 模型参数在线更新，基于近期实际滑点反馈

### 十五、基础设施与部署

**15.1 环境分级**
| 环境 | 数据 | 资金 | 用途 |
|------|------|------|------|
| Paper | 实盘延迟模拟 | 虚拟 | 策略验证、回归测试 |
| Shadow | 实盘数据 | 虚拟 | 生产并行运行，比对信号 |
| Live | 实盘 | 真实 | 全量交易 |

**15.2 配置管理**
- 策略参数：版本化存储（Git LFS 或专用配置服务）
- 运行时热加载：文件 watch 或配置中心推送，变更审计日志
- 环境变量仅用于部署元数据（交易所、账户 ID），策略数值走配置

**15.3 灾备**
- 双活部署：同城双机房，状态实时同步
- 故障转移：心跳超时 3s 触发，RTO < 10s，RPO ≈ 0（WAL 同步）
- 数据备份：tick 数据实时归档至对象存储，保留 1 年


**16.2 混沌测试场景**
- 网络：延迟抖动 10-500ms、随机丢包 1%、连接闪断
- 数据：乱序 tick、重复序列号、时间戳倒跳、字段缺失
- 交易所行为：拒绝所有订单、延迟确认成交、部分成交后拒绝剩余

**16.3 生产影子验证**
- Shadow 环境跑生产数据，信号对比：
  - 一致率 > 99.9% 方可上线
  - 差异分析：数据时间戳、浮点精度、随机数种子

### 十八、Claude 交互深化

**18.1 复杂变更流程**
```
用户：需要支持多账户资金划转
Claude：
1. 【影响分析】→ 涉及模块：accounting, execution, risk
2. 【方案选项】→ 
   A) 内部记账划转（无链上操作，快）
   B) 真实链上/交易所划转（慢，有手续费）
3. 【确认】→ 用户选 A + B 组合方案
4. 【接口设计】→ 出 trait 定义，确认后 impl
5. 【实现】→ 分 3 个 commit：接口、A 实现、B 实现
6. 【验证】→ 提供测试场景：余额不足、划转中撤单、并发划转
```

**18.2 性能优化对话模板**
```
用户：延迟从 5ms 升到 15ms，排查
Claude：
1. 【监控数据请求】→ 请提供最近 1 小时 p99 延迟曲线
2. 【热点假设】→ 可能瓶颈：数据解码、锁竞争、分配
3. 【诊断代码】→ 生成临时插桩，定位到具体函数
4. 【根因】→ 确认是 RwLock 在 tick burst 时排队
5. 【方案】→ 改为 sharded lock 或无锁 ringbuf
6. 【验证】→ benchmark 对比，回测一致性确认
```

**18.3 禁止的简化**
- 不说 "优化一下性能" → 改为 "p99 延迟从 X 降至 Y，CPU 预算 Z"
- 不说 "处理异常" → 改为 "网络超时 T 秒后重试 N 次，退化为 Y 模式"
- 不说 "支持多交易所" → 改为 "统一接口，Binance/OKX 已实现，Bybit 规划中"
